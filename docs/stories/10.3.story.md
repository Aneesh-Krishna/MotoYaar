# Story 10.3 — Accept Invite (New User)

**Epic:** 10 — Vehicle Sharing
**Story:** 10.3
**Status:** Draft
**Created:** 2026-03-15
**Depends On:** Story 10.2 (invite acceptance flow for existing users), Story 2.2 (onboarding flow)

---

## Story

**As a** new user who received a vehicle invite,
**I want** the invite to be automatically applied after I sign up,
**so that** I don't have to click the invite link a second time.

---

## Acceptance Criteria

1. - [ ] User clicks invite link while not logged in → `/invites/[token]` → redirected to `/login?inviteToken=[token]`
2. - [ ] After Google SSO login: invite token preserved through the auth flow
3. - [ ] After completing onboarding (or for returning user after first-time login): pending `inviteToken` is resolved → `VehicleAccess` created → user redirected to the shared vehicle
4. - [ ] Invite token stored in a short-lived cookie during the sign-in flow (cookie name: `pending_invite_token`, max-age 10 minutes)
5. - [ ] If invite expired during sign-up flow: show "The invite link has expired. Ask the vehicle owner to resend it." message, then redirect to `/garage`
6. - [ ] Implementation: store `inviteToken` in a cookie on the `/login` page; read and resolve it in `(app)/layout.tsx` or a dedicated client hook after onboarding completes
7. - [ ] `pnpm typecheck` and `pnpm lint` pass

## Dev Notes

### Cookie Strategy

[Source: architecture.md#5-api-specification]

The invite token must survive the OAuth redirect cycle. The recommended approach:

1. **On `/login?inviteToken=[token]`**: the login page (or a server action) sets a short-lived cookie before redirecting to Google:

```typescript
// In the login page component or a server action
// Set cookie before signIn() call
document.cookie = `pending_invite_token=${inviteToken}; path=/; max-age=600; SameSite=Lax`;
signIn("google", { callbackUrl: "/garage" });
```

Or set it server-side via a response cookie in a Next.js route handler if client-side is unreliable across redirects. Server-side approach is more robust:

```typescript
// src/app/api/auth/set-invite-cookie/route.ts
export async function POST(req: Request) {
  const { token } = await req.json();
  const response = NextResponse.json({ ok: true });
  response.cookies.set("pending_invite_token", token, {
    httpOnly: true,
    maxAge: 600, // 10 minutes
    path: "/",
    sameSite: "lax",
  });
  return response;
}
```

The login page calls this endpoint before calling `signIn()`.

2. **After auth completes**: in `(app)/layout.tsx` (or a client component that wraps it), read the cookie and resolve the pending invite:

```tsx
// src/app/(app)/layout.tsx — add this effect in a client wrapper
"use client";
import { useEffect } from "react";
import { useSession } from "next-auth/react";
import Cookies from "js-cookie"; // or parse manually

export function PendingInviteResolver() {
  const { data: session } = useSession();

  useEffect(() => {
    if (!session) return;
    const pendingToken = Cookies.get("pending_invite_token");
    if (!pendingToken) return;

    // Clear the cookie
    Cookies.remove("pending_invite_token");

    // Resolve the invite
    fetch(`/api/invites/${pendingToken}/accept`, { method: "POST" })
      .then(async res => {
        if (res.ok) {
          const { vehicleId } = await res.json();
          window.location.href = `/garage/${vehicleId}`;
        } else if (res.status === 410) {
          // Expired
          toast.error("The invite link has expired. Ask the vehicle owner to resend it.");
          window.location.href = "/garage";
        }
      })
      .catch(() => {});
  }, [session?.user.id]);

  return null;
}
```

If `js-cookie` is not installed: `pnpm add js-cookie && pnpm add -D @types/js-cookie`.

### POST /api/invites/[token]/accept

[Source: architecture.md#5-api-specification]

Create `src/app/api/invites/[token]/accept/route.ts`:

```typescript
export async function POST(req: Request, { params }: { params: { token: string } }) {
  const session = await getServerSession(authOptions);
  if (!session) return NextResponse.json({ error: "Unauthorized" }, { status: 401 });

  try {
    const result = await vehicleInviteService.acceptInvite(params.token, session.user.id);
    return NextResponse.json({ vehicleId: result.vehicleId });
  } catch (err) {
    if (err instanceof ConflictError && err.message.includes("expired")) {
      return NextResponse.json({ error: "EXPIRED" }, { status: 410 });
    }
    return handleApiError(err);
  }
}
```

Note: `acceptInvite()` is already implemented in Story 10.2. This route calls the same method.

### Login Page — Token Forwarding

[Source: architecture.md#6-components]

Read `src/app/login/page.tsx` (or wherever the login page is). Extend it to:

1. Read `inviteToken` from `searchParams`
2. If present: set the cookie (client-side or via the `/api/auth/set-invite-cookie` endpoint) before calling `signIn()`

```tsx
// In the login page (client component)
const searchParams = useSearchParams();
const inviteToken = searchParams.get("inviteToken");

const handleSignIn = async () => {
  if (inviteToken) {
    // Set short-lived cookie so it survives the OAuth redirect
    await fetch("/api/auth/set-invite-cookie", {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify({ token: inviteToken }),
    });
  }
  signIn("google", { callbackUrl: "/garage" });
};
```

### Onboarding Integration

[Source: architecture.md#6-components]

The `PendingInviteResolver` component should be rendered in `src/app/(app)/layout.tsx`. It runs on every authenticated page load but checks for the cookie first — it is a no-op if no cookie is present. This avoids any performance impact.

If the user is a new user who is still going through onboarding, the `callbackUrl` in `signIn()` is `/garage` which will show the onboarding. The `PendingInviteResolver` runs once the session is active. If onboarding is not yet complete, the user should first complete onboarding, then the resolver runs. To avoid conflicting redirects, only resolve the invite AFTER `user.onboardingCompleted = true`.

```tsx
useEffect(() => {
  if (!session || !session.user.onboardingCompleted) return;
  // ... check cookie and resolve
}, [session?.user.id, session?.user.onboardingCompleted]);
```

### Expiry During Sign-Up

[Source: architecture.md#7-error-handling]

If the 3-day invite expires while the user is still in the sign-up/onboarding flow:
- The `acceptInvite()` method will throw a `ConflictError` with message "expired"
- The API returns 410
- The client shows a Sonner toast with the expiry message
- User is redirected to `/garage` (their now-empty garage)
