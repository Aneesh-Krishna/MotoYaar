# Story 17.7 — Offline: Service Worker Tile Intercept & Client Route Graph

**Epic:** 17 — Maps, Group Rides & Offline Navigation
**Story:** 17.7
**Status:** Draft
**Created:** 2026-05-17
**Depends On:** Story 17.6 (OPFS storage working, tiles stored from download), Story 1.x (Service Worker / PWA already scaffolded via next-pwa)

## Story

**As a** rider in an area with no internet connection,
**I want** the map to load from locally stored tiles and navigate me using the downloaded route graph,
**so that** I can navigate turn-by-turn with full re-routing even without any signal.

---

## Acceptance Criteria

1. - [ ] A **Service Worker** intercepts Mapbox tile requests (`https://api.mapbox.com/v4/...`) and serves them from OPFS when the tile exists in the downloaded corridor
2. - [ ] If a tile is not in OPFS, the request **falls through to the network** transparently — no error shown to user
3. - [ ] The Service Worker is **registered and active** before the "Download for Offline" option is offered to the user — if SW is not active, hide the option with a message "Offline navigation not available in this browser"
4. - [ ] The custom SW tile intercept does not conflict with `next-pwa`'s Workbox config — the offline tile strategy is injected via **Workbox custom inject** pattern, not a blanket cache-all rule
5. - [ ] The downloaded **route graph** (loaded from OPFS) powers navigation decisions: current step, distance to next turn, upcoming instruction
6. - [ ] **Client-side Dijkstra re-routing**: if the rider deviates >150m from the route while offline, re-routing is performed using the downloaded graph within the corridor
7. - [ ] If re-routing destination falls **outside the downloaded corridor**, display: "Outside offline zone — reconnect for rerouting" — do NOT silently fail
8. - [ ] Navigation state during offline mode (current step, position on route, speed) is stored in Zustand and survives component re-mounts
9. - [ ] The offline navigation session **buffers location snapshots** in IndexedDB; on reconnect, they are uploaded to `location_snapshots` (if snapshots enabled)
10. - [ ] `pnpm typecheck` and `pnpm lint` pass

## Dev Notes

### ⚠️ Service Worker & next-pwa Conflict

This is the highest-risk technical task in the offline block. `next-pwa` generates its own Workbox config and registers a SW. Injecting a custom tile intercept strategy requires **Workbox inject manifest** pattern — editing the SW source file that `next-pwa` processes.

The key risk: if Workbox's `GenerateSW` strategy is used (default), you cannot add custom fetch handlers. You must switch to `InjectManifest` strategy in `next.config.js`:

```javascript
// next.config.js
const withPWA = require('next-pwa')({
  dest: 'public',
  swSrc: 'src/sw.ts',           // ← custom SW source (InjectManifest)
  // disable: process.env.NODE_ENV === 'development',
});
```

```typescript
// src/sw.ts (Service Worker source — compiled by Workbox)
import { precacheAndRoute } from 'workbox-precaching';
import { registerRoute } from 'workbox-routing';

// next-pwa injects precache manifest here
precacheAndRoute(self.__WB_MANIFEST);

// Custom tile intercept — runs BEFORE any other route
self.addEventListener('fetch', (event: FetchEvent) => {
  const url = new URL(event.request.url);
  if (url.hostname === 'api.mapbox.com' && url.pathname.includes('/v4/')) {
    event.respondWith(handleTileRequest(event.request));
  }
});

async function handleTileRequest(request: Request): Promise<Response> {
  // Extract z/x/y from URL
  const tileKey = extractTileKey(request.url); // e.g., "12/1234/2345"
  const tile = await readTileFromOPFS(tileKey);
  if (tile) {
    return new Response(tile, {
      headers: { 'Content-Type': 'application/x-protobuf' }
    });
  }
  // Fallthrough to network
  return fetch(request);
}
```

**Test on physical iOS Safari** — SW tile intercept behaviour on iOS Safari has historically been less reliable than Chrome. Do not consider this story done without a physical iOS device test.

### OPFS Tile Key Mapping

Tile keys in OPFS use the Slippy Map convention: `{z}/{x}/{y}.mvt`. When a tile request arrives in the SW, extract the z/x/y from the URL and look up the corresponding OPFS file.

```typescript
// In SW context — OPFS access from Service Worker (not the same OPFS handle as the page)
// SW uses its own navigator.storage.getDirectory() — same OPFS root, accessible from SW
async function readTileFromOPFS(tileKey: string): Promise<ArrayBuffer | null> {
  try {
    const root = await navigator.storage.getDirectory();
    const dir = await root.getDirectoryHandle('offline-routes', { create: false });
    // tileKey format: "routeId/tiles/z/x/y.mvt"
    const [routeId, _, z, x, yMvt] = tileKey.split('/');
    const tilesDir = await dir.getDirectoryHandle(routeId, { create: false });
    const zDir = await tilesDir.getDirectoryHandle(z, { create: false });
    const xDir = await zDir.getDirectoryHandle(x, { create: false });
    const fileHandle = await xDir.getFileHandle(yMvt);
    return (await fileHandle.getFile()).arrayBuffer();
  } catch {
    return null; // tile not in OPFS → fall through to network
  }
}
```

### Offline Snapshot Buffering

During offline mode, location snapshots (if `snapshots_enabled === true`) are buffered in IndexedDB:

```typescript
// src/lib/offline/snapshotBuffer.ts
export async function bufferSnapshot(snapshot: LocationSnapshot): Promise<void> {
  const db = await openSnapshotBufferDb();
  await db.add('pending_snapshots', { ...snapshot, bufferedAt: Date.now() });
}

// Called on network reconnect
export async function flushSnapshots(): Promise<void> {
  const db = await openSnapshotBufferDb();
  const pending = await db.getAll('pending_snapshots');
  if (pending.length === 0) return;
  await fetch('/api/ride-groups/snapshots/bulk', {
    method: 'POST',
    body: JSON.stringify(pending),
  });
  await db.clear('pending_snapshots');
}
```
