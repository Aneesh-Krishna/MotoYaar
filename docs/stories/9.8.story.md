# Story 9.8 — Guest (Unauthenticated) Community Access

**Epic:** 09 — Community
**Story:** 9.8
**Status:** Draft
**Created:** 2026-03-15
**Depends On:** Story 9.2 (community feed), Story 9.3 (post detail)

---

## Story

**As a** visitor who is not signed in,
**I want** to browse community posts and read discussions,
**so that** I can discover the community before deciding to sign up.

---

## Acceptance Criteria

1. - [ ] `/community` and `/community/[id]` routes are NOT protected by the auth middleware (already handled in Story 2.5 — verify middleware matcher excludes them)
2. - [ ] Feed loads and displays posts for unauthenticated visitors (same trending algorithm)
3. - [ ] Like/Dislike/Comment/"New post" interactions: if no session → show `LoginPromptModal`
4. - [ ] `LoginPromptModal`: "Sign in with Google to join the conversation" + Google sign-in button
5. - [ ] `src/components/community/LoginPromptModal.tsx` — client component
6. - [ ] Guest feed uses same `communityService.listPosts()` (no `userId` passed → `userReaction=null` for all posts)
7. - [ ] `GET /api/posts` accepts optional session; returns `userReaction` if authenticated, `null` if not (already handled in `communityService.listPosts`)
8. - [ ] No personal data of other users exposed to guests — avatars and usernames are OK; emails never returned
9. - [ ] `pnpm typecheck` and `pnpm lint` pass

## Dev Notes

### Middleware Verification

[Source: architecture.md#5-api-specification]

Read `src/middleware.ts` (or `middleware.ts` at root) to confirm `/community` and `/community/[id]` are excluded from the auth matcher. The matcher from Story 2.5 should look like:

```typescript
export const config = {
  matcher: [
    "/((?!api|_next/static|_next/image|favicon.ico|community|invites|login|privacy|icons).*)",
  ],
};
```

If `/community` is missing from the exclusion list, update the middleware matcher. Do NOT remove other protected routes.

### Guest State in Feed

[Source: architecture.md#6-components]

The community page RSC tries `getServerSession()`. If no session, it passes `undefined` as `userId`:

```tsx
// src/app/(app)/community/page.tsx
const session = await getServerSession(authOptions);
// session may be null for guests — that is OK

const initialData = await communityService.listPosts("trending", 1, undefined, session?.user.id);

return <CommunityFeedView initialData={initialData} isAuthenticated={!!session} />;
```

`isAuthenticated` is passed to `PostCard` to conditionally render interaction buttons vs login prompt triggers.

### LoginPromptModal Component

[Source: front-end-spec.md#5-component-library--ui-patterns]

Create `src/components/community/LoginPromptModal.tsx`:

```tsx
"use client";
import { signIn } from "next-auth/react";
import { Dialog, DialogContent } from "@/components/ui/dialog";
import { Button } from "@/components/ui/button";

interface LoginPromptModalProps {
  open: boolean;
  onOpenChange: (open: boolean) => void;
  message?: string;
}

export function LoginPromptModal({
  open,
  onOpenChange,
  message = "Sign in with Google to join the conversation",
}: LoginPromptModalProps) {
  return (
    <Dialog open={open} onOpenChange={onOpenChange}>
      <DialogContent className="max-w-sm text-center">
        <div className="py-4">
          <div className="w-12 h-12 bg-orange-100 rounded-full flex items-center justify-center mx-auto mb-4">
            <Users className="text-orange-500" size={24} />
          </div>
          <h2 className="text-lg font-bold text-gray-900 mb-2">Join the conversation</h2>
          <p className="text-sm text-gray-500 mb-6">{message}</p>
          <Button
            onClick={() => signIn("google", { callbackUrl: window.location.href })}
            className="w-full bg-white text-gray-700 border border-gray-300 hover:bg-gray-50"
          >
            <img src="/icons/google.svg" alt="" className="w-4 h-4 mr-2" />
            Continue with Google
          </Button>
        </div>
      </DialogContent>
    </Dialog>
  );
}
```

The `callbackUrl` is set to the current page so the user returns to where they were after signing in.

### Interaction Guard Pattern

[Source: architecture.md#6-components]

In `PostCard` and `PostDetailView`, guard interactions:

```tsx
// In PostCard
const [showLoginPrompt, setShowLoginPrompt] = useState(false);

const handleLike = () => {
  if (!isAuthenticated) {
    setShowLoginPrompt(true);
    return;
  }
  // actual like logic
};

// ...
<button onClick={handleLike}>👍 {post.likes}</button>
{showLoginPrompt && (
  <LoginPromptModal
    open={showLoginPrompt}
    onOpenChange={setShowLoginPrompt}
  />
)}
```

Apply the same pattern for:
- Like button
- Dislike button
- Comment input (show prompt instead of focusing input)
- "New post" FAB (show prompt instead of navigating)

### No Sensitive Data Exposure

[Source: architecture.md#5-api-specification]

The `GET /api/posts` and `GET /api/posts/[id]` routes must never return user email addresses. Review the Drizzle `with: { user: ... }` selections:

```typescript
with: {
  user: {
    columns: {
      id: true,
      name: true,
      image: true,
      // email: NEVER included
    },
  },
}
```

Confirm this is the case in all community service queries.

---

### "New Post" FAB Guard

[Source: front-end-spec.md#3-screen-by-screen-specifications]

In the community page, the FAB (Floating Action Button) navigates to `/community/new`. For guests, show `LoginPromptModal` instead:

```tsx
const handleNewPost = () => {
  if (!isAuthenticated) {
    setShowLoginPrompt(true);
    return;
  }
  router.push("/community/new");
};
```
