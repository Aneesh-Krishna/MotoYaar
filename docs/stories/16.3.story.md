## Story

**As a** member of a group expense session,
**I want** to see who owes what to whom and mark debts as settled,
**so that** we can cleanly resolve the money owed after our group trip without confusion.

---

## Acceptance Criteria

1. - [ ] A **"Balances"** tab or section within the group session shows the net amount each member owes or is owed
2. - [ ] Balances are displayed as simplified debts: e.g. *"Priya owes Aneesh ₹340"* — chains are collapsed (A owes B ₹100, B owes C ₹100 → A owes C ₹100)
3. - [ ] From the current user's perspective, their view is personalised: *"You owe Rahul ₹220"* / *"Vikram owes you ₹180"*
4. - [ ] The session also shows a summary: **total session spend**, **per-member total share**
5. - [ ] The creator or either party in a debt can tap **"Mark as Settled"** on a specific balance entry — this records the settlement and removes the debt from the active balance view
6. - [ ] Settled debts are shown in a collapsible **"Settled"** section below active balances (for record keeping)
7. - [ ] When all balances in a session are settled, a banner is shown: *"All settled! 🎉"* and the creator can archive the session
8. - [ ] Balances are recalculated live whenever a new expense is added or edited
9. - [ ] `pnpm typecheck` and `pnpm lint` pass

## Dev Notes

### Balance Calculation Algorithm

Standard debt simplification (same as Splitwise):

```typescript
function computeSimplifiedBalances(items: GroupExpenseItem[], members: string[]): Balance[] {
  // Step 1: Compute net balance per member
  const net: Record<string, number> = {};
  members.forEach(m => net[m] = 0);

  for (const item of items) {
    const share = item.amount / item.includedUserIds.length;
    // Payer is owed money from included members
    net[item.paidBy] += item.amount;
    // Each included member owes their share
    item.includedUserIds.forEach(uid => {
      net[uid] -= share;
    });
  }

  // Step 2: Simplify into minimal transactions
  const creditors = Object.entries(net).filter(([, v]) => v > 0).sort((a, b) => b[1] - a[1]);
  const debtors   = Object.entries(net).filter(([, v]) => v < 0).sort((a, b) => a[1] - b[1]);

  const transactions: Balance[] = [];
  let i = 0, j = 0;
  while (i < creditors.length && j < debtors.length) {
    const [credId, credAmt] = creditors[i];
    const [debtId, debtAmt] = debtors[j];
    const amount = Math.min(credAmt, Math.abs(debtAmt));
    transactions.push({ from: debtId, to: credId, amount: parseFloat(amount.toFixed(2)) });
    creditors[i][1] -= amount;
    debtors[j][1]  += amount;
    if (Math.abs(creditors[i][1]) < 0.01) i++;
    if (Math.abs(debtors[j][1])  < 0.01) j++;
  }
  return transactions;
}
```

This runs server-side on `GET /api/group-expenses/[id]/balances` and is not persisted — always computed fresh from current items.

### Settlement DB Schema

```sql
CREATE TABLE group_expense_settlements (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  session_id   UUID NOT NULL REFERENCES group_expense_sessions(id) ON DELETE CASCADE,
  from_user_id UUID NOT NULL REFERENCES users(id),
  to_user_id   UUID NOT NULL REFERENCES users(id),
  amount       NUMERIC(12,2) NOT NULL,
  settled_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
  settled_by   UUID NOT NULL REFERENCES users(id)  -- who marked it settled
);
```

The balance computation excludes settled amounts: when computing balances, settled amounts are subtracted from the relevant items first.

#### Findings

**F1 — MINOR — Empty-session "All settled!" false positive**
- **Location:** `BalancesPanel.tsx:32`, `computeBalances()` service
- **Issue:** `allSettled = data.activeBalances.length === 0` evaluates to `true` when a session has zero expense items (no items → no net balances → no active balances → banner fires). A brand-new session with no expenses shows "All settled! 🎉" and an archive CTA.
- **Risk:** Low — confusing UX for session creator before any expenses are logged.
- **Recommendation:** Guard with `data.totalSessionSpend > 0 && data.activeBalances.length === 0`, or show a neutral empty state when no items exist yet.

**F2 — CONCERN — `settleBalance` accepts arbitrary amounts without validating against actual balance**
- **Location:** `groupExpenseService.ts:607`, `settlements/route.ts`
- **Issue:** The POST endpoint accepts any positive `amount` without checking that it matches an actual outstanding balance entry. A user can POST `{ fromUserId, toUserId, amount: 0.01 }` against a ₹500 debt and it records successfully, partially reducing the net. Multiple small settlements can fragment a single debt. The `computeBalances` computation does correctly net these out, but the UX intent (from AC5: "removes the debt from the active balance view") implies settling the full computed amount.
- **Risk:** Medium — produces confusing partial settlements; can result in outstanding micro-balances (e.g., ₹0.01) that never clear.
- **Recommendation:** Validate that `input.amount` is within a reasonable tolerance of the actual computed balance between `fromUserId` and `toUserId` at settlement time, OR document that partial settlements are intentional. If intentional, the UI should reflect this (e.g., show remaining balance after partial settle).

- **Risk:** Very Low — minor DB overhead on every visit. Acceptable for current traffic.

#### Round 2 Findings

**R2-F1 — CONCERN — Settlement amount validation uses incorrect approximation in multi-creditor sessions**
- **Location:** `groupExpenseService.ts:647–655`
- **Issue:** The validation computes `outstandingDebt = Math.min(|fromNet|, max(0, toNet))` using raw net balances, not the Splitwise-simplified pairwise amount. This breaks when a session has multiple creditors:

  *Example:* A pays ₹200 for [B,C]; D pays ₹100 for [B].
  Net: A=+200, B=-200, C=-100, D=+100.
  Simplified transactions (by algorithm): B→A ₹100, C→A ₹100, B→D ₹100.
  UI shows B owes A **₹100** and B owes D ₹100.
  B taps "Mark as Settled" for the B→A ₹100 entry.
  Service receives `{ fromUserId: B, toUserId: A, amount: 100 }`.
  `fromNet[B] = -200`, `toNet[A] = +200` → `outstandingDebt = min(200, 200) = 200`.
  `|100 - 200| = 100 > 1` → **BadRequestError thrown** — settlement rejected incorrectly.

- **Risk:** High in practice — any session with ≥3 members where one person has multiple debts will fail to settle the correct UI-displayed amount.
- **Recommendation:** The correct approach is to run the full Splitwise simplification algorithm (already implemented in `computeBalances`) within `settleBalance` and find the specific pairwise balance between `fromUserId` and `toUserId` in the simplified result. Then validate `input.amount` is within tolerance of that simplified amount. This is the same computation already available — just needs to be applied here rather than the raw-net approximation.

**CONCERNS** — F1 and F4 resolved cleanly. F2 partial fix introduced a regression: the approximation actively breaks valid settlements in real multi-member group scenarios (which is the primary use case). The fix needs one more iteration.
