# Story 4.2 — Manual Expiry Entry Fallback

**Epic:** 04 — Document Management
**Story:** 4.2
**Status:** Draft
**Created:** 2026-03-15
**Depends On:** Story 4.1 (Document Upload flow — manual entry is a fallback screen within it)

---

## Story

**As a** user whose document couldn't be parsed,
**I want** to manually enter the expiry date,
**so that** my document is still tracked even when AI extraction fails.

---

## Acceptance Criteria

1. - [ ] Manual entry screen is shown when: AI parsing returns null `expiryDate`; OR user taps "Enter date manually instead" on the AI confirmation screen (Story 4.1 Screen 3)
2. - [ ] Manual entry screen shows a date picker (shadcn/ui `Calendar` inside a `Popover`) for selecting the expiry date
3. - [ ] "Skip" option is available — saves document record with `parseStatus = "incomplete"` and `expiryDate = null`
4. - [ ] On save with a date selected: document saved with `parseStatus = "manual"` and the user-entered `expiryDate`
5. - [ ] Incomplete documents (`parseStatus = "incomplete"`, `expiryDate = null`) appear in the Documents tab with a gray `StatusBadge` ("Incomplete") and a note "Tap to add expiry date"
6. - [ ] User can return later and add the expiry date via document edit (Story 4.5)
7. - [ ] The manual entry screen appears within the same `BottomSheet` flow started in Story 4.1 (same sheet, new screen state — no new sheet opened)
8. - [ ] `pnpm typecheck` and `pnpm lint` pass

```typescript
type UploadScreen = "upload" | "parsing" | "confirm" | "manual" | "complete";
```

When transitioning to `"manual"`:

### Manual Entry Screen UI

[Source: front-end-spec.md#5-component-library--ui-patterns]

```tsx
// Inside DocumentUpload, when screen === "manual"
<div className="flex flex-col gap-6 p-4">
  <div className="text-center">
    <h3 className="font-semibold text-gray-800">Enter Expiry Date</h3>
    <p className="text-sm text-gray-500 mt-1">
      {parseStatus === "failed"
        ? "We couldn't read the date from your document."
        : "Enter the date manually."}
    </p>
  </div>

  {/* shadcn Calendar in Popover */}
  <div className="flex flex-col gap-2">
    <label className="text-sm font-medium text-gray-700">Expiry Date</label>
    <Popover open={calendarOpen} onOpenChange={setCalendarOpen}>
      <PopoverTrigger asChild>
        <button className={cn(
          "flex items-center gap-2 border rounded-lg px-3 py-2 text-sm w-full",
          !selectedDate && "text-gray-400"
        )}>
          <CalendarIcon size={16} />
          {selectedDate ? format(selectedDate, "d MMMM yyyy") : "Select date"}
        </button>
      </PopoverTrigger>
      <PopoverContent className="w-auto p-0">
        <Calendar
          mode="single"
          selected={selectedDate}
          onSelect={(date) => { setSelectedDate(date); setCalendarOpen(false); }}
          initialFocus
          fromDate={new Date()} // optional — allow past dates too; remove if docs can be already expired
        />
      </PopoverContent>
    </Popover>
  </div>

  <button
    onClick={handleManualSave}
    disabled={!selectedDate || isSaving}
    className="bg-orange-500 text-white w-full py-3 rounded-lg font-semibold disabled:opacity-50"
  >
    {isSaving ? <Loader2 className="animate-spin mx-auto" size={20} /> : "Save Document"}
  </button>

  <button
    onClick={handleSkip}
    className="text-gray-500 text-sm underline text-center"
  >
    Skip for now
  </button>
</div>
```

Note: The Calendar `fromDate` constraint is intentionally relaxed — users may be entering an already-expired document (e.g. they had insurance that expired and are logging it). Past dates should be allowed.

### Save Handlers

[Source: architecture.md#5-api-specification]

Two distinct save paths from the manual entry screen:

**Save with date (`handleManualSave`):**
```typescript
const handleManualSave = async () => {
  if (!selectedDate) return;
  setIsSaving(true);
  try {
    await documentApi.createDocument(vehicleId, {
      type: docType,
      label: docLabel,
      expiryDate: format(selectedDate, "yyyy-MM-dd"),
      parseStatus: "manual",
      tempR2Key: tempR2Key ?? undefined,
    });
    toast.success("Document saved");
    onClose();
  } catch (error) {
    toast.error("Failed to save document. Try again.");
  } finally {
    setIsSaving(false);
  }
};
```

**Skip (`handleSkip`):**
```typescript
const handleSkip = async () => {
  setIsSaving(true);
  try {
    await documentApi.createDocument(vehicleId, {
      type: docType,
      label: docLabel,
      expiryDate: undefined,     // null in DB
      parseStatus: "incomplete",
      tempR2Key: tempR2Key ?? undefined,
    });
    toast.success("Document saved as incomplete");
    onClose();
  } catch (error) {
    toast.error("Failed to save document. Try again.");
  } finally {
    setIsSaving(false);
  }
};
```

Both handlers call the same `POST /api/vehicles/[id]/documents` endpoint (Story 4.1). The `parseStatus` field determines how `documentService.create()` handles storage.
