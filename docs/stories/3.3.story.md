## Story

**As a** user,
**I want** to add a vehicle through a guided six-step wizard,
**so that** I can set up my vehicle's complete profile without feeling overwhelmed by a long form.

---

## Acceptance Criteria

1. - [ ] Route: `src/app/(app)/garage/new/page.tsx` — full-screen step wizard (not a modal)
2. - [ ] Progress bar visible throughout: "Step X of 6"
3. - [ ] Back button available on steps 2–6; step 1 has a close/cancel button (returns to `/garage`)
4. - [ ] Step 1: Vehicle Name (required text input) + Vehicle Type dropdown (`2-wheeler` / `4-wheeler` / `Truck` / `Other`)
5. - [ ] Step 2: Company (optional), Model (optional), Variant (optional), Color (optional), Registration Number (required)
6. - [ ] Step 3: Purchase Date (date picker, optional), Previous Owners (number input, default `0`, min `0`)
7. - [ ] Step 4: Vehicle image upload (optional) — pre-signed R2 URL upload; image preview shown after upload
8. - [ ] Step 5: Document upload — shows "Add documents later" button and "Skip for now" option; document parsing deferred to Epic 04; if user taps "Add documents later", proceed to step 6
9. - [ ] Step 6: Review screen showing all entered data grouped by section (Steps 1–4); "Edit" link next to each section returns to that step; "Save Vehicle" CTA at bottom
10. - [ ] Registration number duplicate check is server-side: if duplicate found on save, show 409 Conflict error inline ("You already have a vehicle with this registration number")
11. - [ ] On successful save: vehicle record created in DB; user redirected to `/garage/[id]`
12. - [ ] Image upload uses pre-signed R2 PUT URL pattern (same as profile image in Story 2.2)
13. - [ ] Zod schema `createVehicleSchema` in `src/lib/validations/vehicle.ts` validates the final payload
14. - [ ] `pnpm typecheck` and `pnpm lint` pass

### Step 4 — Vehicle Image

[Source: architecture.md#7-external-apis, architecture.md#8-core-workflows]

Image upload uses the pre-signed R2 URL pattern:

1. User taps "Upload Photo" → opens file picker (accept `image/jpeg,image/png,image/webp`, max 5MB)
2. Client calls `POST /api/uploads/vehicle-image` → server returns `{ uploadUrl: string, key: string }` (pre-signed S3 PUT URL)
3. Client does a direct `PUT` to the pre-signed URL with the image binary
4. On success, set `imageUrl` to the public/CDN URL and `imageKey` to the key

API route `POST /api/uploads/vehicle-image`:
```typescript
// Returns pre-signed PUT URL for R2 vehicle image upload
// R2 key: {userId}/vehicles/images/{uuid}.{ext}
// Content-Type: image/*
// Expires: 5 minutes
```

Show image preview after upload using `<Image src={imageUrl} .../>`. "Step is optional" — show "Skip" to proceed without image.

If the user goes back from Step 5+ and changes the image: delete the previously uploaded R2 key before uploading a new one (call `DELETE /api/uploads/vehicle-image?key={imageKey}`).

### API Route: POST /api/vehicles

[Source: architecture.md#5-api-specification]

Create `src/app/api/vehicles/route.ts`:

```typescript
export async function POST(req: Request) {
  const session = await getServerSession(authOptions);
  if (!session) return NextResponse.json({ error: "Unauthorized" }, { status: 401 });

  try {
    const body = await req.json();
    const data = createVehicleSchema.parse(body);
    const vehicle = await vehicleService.create(session.user.id, data);
    return NextResponse.json(vehicle, { status: 201 });
  } catch (error) {
    return handleApiError(error);
  }
}
```

Route also needs `GET` (list vehicles) but that can be a stub returning `[]` until Story 3.2 service methods are implemented.

### vehicleService.create()

[Source: architecture.md#6-components]

Add to `src/services/vehicleService.ts`:

```typescript
async create(userId: string, data: CreateVehicleInput): Promise<Vehicle> {
  // 1. Check for duplicate registration number for this user
  const existing = await db.query.vehicles.findFirst({
    where: and(
      eq(vehicles.userId, userId),
      eq(vehicles.registrationNumber, data.registrationNumber.toUpperCase())
    )
  });
  if (existing) throw new ConflictError("You already have a vehicle with this registration number");

  // 2. Insert vehicle
  const [vehicle] = await db.insert(vehicles).values({
    userId,
    ...data,
    registrationNumber: data.registrationNumber.toUpperCase(),
  }).returning();

  return vehicle;
}
```

### Zod Schema

[Source: architecture.md#17-coding-standards]

Create `src/lib/validations/vehicle.ts`:

```typescript
import { z } from "zod";

export const createVehicleSchema = z.object({
  name: z.string().min(2, "Name must be at least 2 characters"),
  type: z.enum(["2-wheeler", "4-wheeler", "truck", "other"]),
  company: z.string().optional(),
  model: z.string().optional(),
  variant: z.string().optional(),
  color: z.string().optional(),
  registrationNumber: z.string().min(1, "Registration number is required"),
  purchasedAt: z.string().optional(),        // ISO date string
  previousOwners: z.number().int().min(0).default(0),
  imageUrl: z.string().url().optional(),
  imageKey: z.string().optional(),
});

export const updateVehicleSchema = createVehicleSchema.partial().extend({
  registrationNumber: z.string().min(1).optional(),
});

export type CreateVehicleInput = z.infer<typeof createVehicleSchema>;
export type UpdateVehicleInput = z.infer<typeof updateVehicleSchema>;
```

### Error Handling

[Source: architecture.md#18-error-handling-strategy]

On the Review step save:
- `409 Conflict` → show inline error below Reg Number section: "You already have a vehicle with this registration number"
- `422 Validation` → highlight offending fields (unlikely at this stage but handle gracefully)
- Network error → toast via Sonner: "Failed to save vehicle. Try again."

Use `handleApiError` on the server; use `ApiError` from `src/lib/api-client.ts` on the client to detect error codes.
