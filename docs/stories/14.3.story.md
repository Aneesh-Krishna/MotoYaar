# Story 14.3 — Service Reminders (km + date-based)

## Story

**As a** vehicle owner,
**I want** to set service reminders for my vehicle (oil change, chain lube, tyre check),
**so that** MotoYaar alerts me when a service is due — either by km driven or by elapsed time.

---

## Acceptance Criteria

1. - [ ] A **"Service Reminders"** section is accessible from the Vehicle Detail page (kebab menu → "Manage Reminders" or a dedicated tab / sub-section)
2. - [ ] User can add a reminder with: **Service type** (dropdown with smart defaults + custom option), **km interval** (optional), **date interval in days** (optional). At least one interval must be provided.
3. - [ ] Smart default service types pre-loaded as quick-add chips (user taps to add with default intervals): Oil Change, Chain Lubrication, Air Filter, Tyre Pressure Check, General Service. User can customise intervals after adding.
4. - [ ] Smart default intervals per vehicle type are pre-filled when a chip is selected (see §4.2 in outcomes doc for the defaults table). User can override.
5. - [ ] When a fuel expense with odometer is logged (Story 14.2), the system checks all km-interval reminders for that vehicle and flags any that are due
6. - [ ] The daily cron job (`/api/cron/document-expiry` or a new `/api/cron/service-reminders`) checks date-interval reminders for all users daily
7. - [ ] **Graceful degradation:** If no odometer data is available for a vehicle, km-based reminders are shown as *"Due in ~X km (no odometer data yet)"* — no notification sent until first odometer reading logged
8. - [ ] Reminder notification type: *"Your [vehicle name] is due for [service type]"* — in-app bell + PWA push (same pipeline as document expiry)
9. - [ ] User can delete a reminder from the manage reminders screen
10. - [ ] User can mark a reminder as "Serviced now" — resets the km/date countdown from the current point
11. - [ ] `pnpm typecheck` and `pnpm lint` pass

## Dev Notes

### DB Schema

```sql
CREATE TABLE service_reminders (
  id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  vehicle_id       UUID NOT NULL REFERENCES vehicles(id) ON DELETE CASCADE,
  user_id          UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  service_type     TEXT NOT NULL,           -- "Oil Change", "Chain Lubrication", custom text
  km_interval      INTEGER,                 -- null if date-only reminder
  day_interval     INTEGER,                 -- null if km-only reminder
  last_serviced_km INTEGER,                 -- odometer at last service (null if not tracked)
  last_serviced_at DATE,                    -- date of last service
  notified_at      TIMESTAMPTZ,             -- last notification sent; prevents duplicate alerts
  created_at       TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_service_reminders_vehicle_id ON service_reminders(vehicle_id);
CREATE INDEX idx_service_reminders_user_id ON service_reminders(user_id);
```

### Smart Defaults (Pre-loaded Chips)

```typescript
export const SERVICE_REMINDER_DEFAULTS = [
  { type: 'Oil Change',           kmInterval: 5000, dayInterval: 180 },
  { type: 'Chain Lubrication',    kmInterval: 500,  dayInterval: 30  },
  { type: 'Air Filter',           kmInterval: 6000, dayInterval: 180 },
  { type: 'Tyre Pressure Check',  kmInterval: null, dayInterval: 30  },
  { type: 'General Service',      kmInterval: 10000, dayInterval: 365 },
];
```

For 2-wheelers (type = '2-wheeler'), defaults are as above. For 4-wheelers (type = '4-wheeler'), Oil Change default is 10,000 km / 12 months. Vehicle type is available from the `vehicles` table.

### Cron Check Logic

Add to daily cron (or create separate cron endpoint):
```typescript
// For each service_reminder:
//   km-based: if last_serviced_km is set AND latest odometer from expenses >= (last_serviced_km + km_interval)
//             AND notified_at is null or > dayInterval days ago → send notification
//   date-based: if last_serviced_at + day_interval <= today
//              AND notified_at is null or > 7 days ago → send notification
```

### "Serviced Now" Action

Updates `last_serviced_km` (from latest fuel expense odometer for the vehicle, if available), `last_serviced_at` (today), and clears `notified_at`. This resets the countdown.

### Summary
Implementation is architecturally sound and passes all automated checks (typecheck, lint, 615 tests passing). All ACs are addressed. One functional correctness bug must be fixed before merge.

### Required Fix (Blocks Merge)

**F3 — CRITICAL: Km-based baseline seeding never executes**
- **File:** `src/services/serviceReminderService.ts`, `checkKmRemindersForVehicle()`
- **Bug:** `eq(serviceReminders.lastServicedKm, null as unknown as number)` generates SQL `= NULL` which always evaluates false in Postgres. The baseline update never runs.
- **Impact:** Newly created km-only reminders will never establish an odometer baseline, so km-based notifications will never fire until the user manually taps "Serviced now" first.
- **Fix:** Replace with `isNull(serviceReminders.lastServicedKm)` from drizzle-orm.

### Recommended Fixes (Non-blocking)

**F1 — MEDIUM: PWA push missing from notification path**
- AC8 specifies in-app bell + PWA push. Only `notificationService.create()` (in-app) is called. `pushService` is not invoked.
- Fix: Add `pushService.getSubscriptionsForUser()` + `sendPushNotification()` in `runDateReminderCheck()`, mirroring `cronService.runDocumentExpiryCheck()`.

**F5 — LOW: Silent UI errors on delete/serviced-now**
- `handleDelete()` optimistically removes item from state with no error handling on failure.
- `handleServiced()` silently does nothing on API error.
- Fix: Add error state/toast feedback on failure.

### Informational Notes
- **F2:** Shared-vehicle viewers see empty reminders panel (owner's reminders filtered by userId) — correct privacy behaviour but no explanatory message shown.
- **F4:** `formatDue()` shows only km interval for dual-interval reminders; day interval silently omitted from display string.
