# Story 4.3 — Document List with Expiry Status

**Epic:** 04 — Document Management
**Story:** 4.3
**Status:** Draft
**Created:** 2026-03-15
**Depends On:** Story 4.1 (documents can be created before listing is useful)

## Story

**As a** vehicle owner,
**I want** to see all my vehicle's documents with clear expiry status indicators,
**so that** I can tell at a glance which documents need attention.

---

## Acceptance Criteria

1. - [ ] The Documents tab on the vehicle detail page lists all documents for that vehicle
2. - [ ] `DocumentRow` component (`src/components/documents/DocumentRow.tsx`) renders each document
3. - [ ] Each row shows: document type icon, type label, expiry date formatted as "DD MMM YYYY", `StatusBadge`
4. - [ ] `StatusBadge` colors per status: Green (valid, >30 days until expiry), Amber (within `user.notificationWindowDays`, default 30), Red (expired or expiry date is today), Gray (incomplete — no expiry date)
5. - [ ] Status is computed server-side in `documentService.listByVehicle()` — not calculated in the UI component
6. - [ ] Documents sorted: soonest expiry date first; incomplete documents (null expiry date) shown last
7. - [ ] Each row has contextual actions: if `storageUrl` is set and user has `full_storage` preference → "View" button (opens signed URL); "Replace" option available; for all documents → "Edit" and "Delete" (kebab per row)
8. - [ ] "Add document" CTA always visible below the list (opens Story 4.1 upload flow)
9. - [ ] Empty state: "No documents yet. Add your RC, Insurance, and PUC to get expiry alerts." with "Add document" CTA
10. - [ ] `GET /api/vehicles/[id]/documents` route returns documents with computed status
11. - [ ] `documentService.listByVehicle(vehicleId, userId)` is used by the page; ownership/access check included
12. - [ ] `pnpm typecheck` and `pnpm lint` pass

### documentService.listByVehicle()

[Source: architecture.md#6-components]

Add to `src/services/documentService.ts`:

```typescript
async listByVehicle(vehicleId: string, userId: string): Promise<Document[]> {
  // Access check (owner or viewer)
  await vehicleService.getWithAccessCheck(vehicleId, userId);

  // Fetch user for notificationWindowDays
  const user = await db.query.users.findFirst({ where: eq(users.id, userId) });
  const windowDays = user?.notificationWindowDays ?? 30;

  const docs = await db.query.documents.findMany({
    where: and(
      eq(documents.vehicleId, vehicleId),
      isNull(documents.deletedAt)  // if soft delete added; otherwise omit
    ),
    orderBy: [
      // Soonest expiry first; NULLs last
      asc(documents.expiryDate),    // NULLs sort last in PostgreSQL ASC
    ],
  });

  // Recompute status for each document (handles cases where cron hasn't run yet)
  return docs.map((doc) => ({
    ...doc,
    status: computeDocumentStatus(
      doc.expiryDate ? doc.expiryDate.toISOString().split("T")[0] : null,
      windowDays
    ),
  }));
}
```

PostgreSQL sorts `NULL` values last in `ORDER BY ... ASC` by default — this satisfies AC 6 without extra logic.

### Contextual Action: View Document

[Source: architecture.md#7-external-apis (Cloudflare R2)]

When user taps "View Document", the client should fetch a signed URL from the server (NOT use the stored URL directly — it is not publicly accessible):

```typescript
// In document row or documents tab component
const handleViewDocument = async (docId: string) => {
  const { signedUrl } = await documentApi.getSignedUrl(docId);
  window.open(signedUrl, "_blank");
};
```

Frontend API wrapper in `src/services/api/documentApi.ts`:
```typescript
export async function getSignedUrl(docId: string): Promise<{ signedUrl: string }> {
  return apiRequest<{ signedUrl: string }>(`/documents/${docId}/url`);
}
```

This route stub can be created here; the signed URL generation uses `generateAccessUrl()` from `src/lib/r2.ts` (Story 1.3 implementation).
