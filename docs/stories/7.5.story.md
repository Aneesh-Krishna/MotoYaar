# Story 7.5 — AI Report Notification Delivery

**Epic:** 07 — Reports & Analytics
**Story:** 7.5
**Status:** Draft
**Created:** 2026-03-15
**Depends On:** Story 7.3 (AI report generation completes), Story 8.3 (email notifications), Story 8.2 (push notifications)

---

## Story

**As a** user who requested an AI report,
**I want** to be notified when my report is ready,
**so that** I can read it without having to manually poll the app.

---

## Acceptance Criteria

1. - [ ] When `ai_report.status` transitions to `'ready'`: an in-app notification is created for the user
2. - [ ] In-app notification: `type="ai_report_ready"`, `title="Your AI report is ready"`, `body="Your spending report for [period] is ready to view."`, `action_url="/reports/ai/[id]"`
3. - [ ] An email is sent to the user via `sendEmail()` from `src/lib/resend.ts`: subject "Your MotoYaar AI report is ready", HTML body includes a link to `/reports/ai/[id]`
4. - [ ] If user has push notifications enabled: a PWA push notification is sent via `sendPushNotification()` from `src/lib/push.ts`
5. - [ ] Notification logic lives in `aiReportService.ts`, called after successful `generateReport()` completes
6. - [ ] `notificationService.create(userId, type, title, body, actionUrl)` used to create in-app notification
7. - [ ] Push send failure (e.g. 410 Gone from push endpoint) does not throw — it is caught silently; stale subscription is deleted from DB
8. - [ ] Email send failure is logged but does not throw — the report is still marked `ready` regardless
9. - [ ] `pnpm typecheck` and `pnpm lint` pass

## Dev Notes

### Notification Trigger Location

[Source: architecture.md#6-components]

The notification logic is called from `aiReportService.runGeneration()` immediately after the `ai_report` record is updated to `status='ready'`:

```typescript
// In aiReportService.runGeneration() (Story 7.3)
await db.update(aiReports)
  .set({ status: "ready", content, completedAt: new Date() })
  .where(eq(aiReports.id, reportId));

// Deliver notifications — errors are non-fatal
await deliverAIReportNotifications(userId, reportId, report.periodLabel);
```

`deliverAIReportNotifications` is a private helper within `aiReportService.ts`:

```typescript
async function deliverAIReportNotifications(
  userId: string,
  reportId: string,
  periodLabel: string
): Promise<void> {
  const actionUrl = `/reports/ai/${reportId}`;

  // 1. In-app notification
  await notificationService.create(
    userId,
    "ai_report_ready",
    "Your AI report is ready",
    `Your spending report for ${periodLabel} is ready to view.`,
    actionUrl
  );

  // 2. Email (non-fatal)
  try {
    const user = await userService.getById(userId);
    await sendEmail(
      user.email,
      "Your MotoYaar AI report is ready",
      buildAIReportReadyEmailHtml(periodLabel, actionUrl)
    );
  } catch (err) {
    logger.warn({ err, userId, reportId }, "AI report email delivery failed");
  }

  // 3. Push (non-fatal, handles stale subscriptions)
  try {
    const subscriptions = await pushService.getSubscriptionsForUser(userId);
    await Promise.allSettled(
      subscriptions.map(sub =>
        sendPushNotification(sub, {
          title: "Your AI report is ready",
          body: `Spending report for ${periodLabel} is ready.`,
          url: actionUrl,
        }).catch(async err => {
          if (err.statusCode === 410) {
            // Stale subscription — delete it
            await pushService.deleteSubscription(sub.endpoint);
          }
        })
      )
    );
  } catch (err) {
    logger.warn({ err, userId }, "AI report push delivery failed");
  }
}
```

### notificationService.create()

[Source: architecture.md#6-components]

`notificationService` lives in `src/services/notificationService.ts`. The `create()` method:

```typescript
async create(
  userId: string,
  type: string,
  title: string,
  body: string,
  actionUrl: string
): Promise<Notification> {
  const [notification] = await db.insert(notifications).values({
    userId,
    type,
    title,
    body,
    actionUrl,
    isRead: false,
    createdAt: new Date(),
  }).returning();
  return notification;
}
```

The `notifications` table must exist (created as part of Story 8.4 schema work). If creating this notification in Story 7.5 before 8.4 is implemented, create the table in this story's migration and note the dependency.

### DB Schema: notifications Table

[Source: architecture.md#3-database-schema]

```sql
CREATE TABLE notifications (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  type        TEXT NOT NULL,           -- 'ai_report_ready' | 'document_expiry' | 'access_revoked' etc.
  title       TEXT NOT NULL,
  body        TEXT NOT NULL,
  action_url  TEXT,
  is_read     BOOLEAN NOT NULL DEFAULT false,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_notifications_user_unread ON notifications(user_id, is_read) WHERE is_read = false;
```

This table is shared with Story 8.4. Create it here if it does not already exist. Check migration history before running.

### Stale Subscription Handling

[Source: architecture.md#11-integrations]

When a push endpoint returns HTTP 410 (Gone), the subscription is no longer valid. The handler must:
1. Catch the 410 error
2. Call `pushService.deleteSubscription(endpoint)` to remove from `push_subscriptions` table
3. Continue processing other subscriptions

`Promise.allSettled` is used so one failing push does not abort others.
