## Story

**As a** rider preparing to ride into an area with poor cell coverage,
**I want** to initiate an offline route download for my planned route,
**so that** the server-side pipeline generates my map tiles, route graph, and voice clips ready for offline use.

## Acceptance Criteria

1. - [ ] A **"Download for Offline"** button appears on the map when a route A→B is active
2. - [ ] Before download, a preview panel shows: estimated download size (MB), coverage area (bounding box on map), expiry date (30 days from today)
3. - [ ] Confirming the download triggers a **Supabase Edge Function** (`/offline/prepare`) that begins the preparation pipeline

8. - [ ] OSRM is **self-hosted** (not using the public API) — do not launch without a self-hosted OSRM instance; document the VPS setup in Dev Agent Record
9. - [ ] The corridor download indicator (download progress) appears as a **persistent bottom bar** on the map page, dismissible once complete
10. - [ ] The `offline_routes` table receives a record on download initiation; the Edge Function updates it on completion
11. - [ ] A **Mapbox API billing account** and access token are confirmed configured before this story deploys (document in Dev Agent Record)

## Dev Notes

### ⚠️ OSRM SELF-HOSTING REQUIREMENT (Non-Negotiable)

The public OSRM API (`router.project-osrm.org`) has no SLA and no uptime commitment. A failure means every offline download fails. Per PRD Section 16:

**Self-host OSRM on a small VPS from day one of offline feature launch — do not launch Story 17.5+ relying on the public API.**

Setup (estimated):
- DigitalOcean or Hetzner VPS, ~$10–20/month
- India OSM extract from [Geofabrik](https://download.geofabrik.de/asia/india.html)
- OSRM Docker: `docker run -t -v $(pwd):/data osrm/osrm-backend osrm-extract -p /opt/car.lua /data/india-latest.osm.pbf`
- Expose HTTP endpoint, restrict to Edge Function IP

Document the VPS endpoint as `OSRM_ENDPOINT` environment variable.

[Source: PRD Section 16 Risks — OSRM SLA risk, PRD Section 11]

### Edge Function Architecture

The `/offline/prepare` Supabase Edge Function runs server-side (Deno). It is the most complex single component in the offline pipeline:

```typescript
// supabase/functions/offline-prepare/index.ts (Deno)
// Steps:
// 1. Decode route polyline from request body
// 2. Compute 500m corridor buffer via Turf.js
// 3. Select Mapbox tile coordinates for zoom 8–15 within bbox
// 4. Call OSRM route graph generation for the corridor
// 5. Call Google Cloud TTS for each turn instruction
// 6. Package → upload to R2 → generate pre-signed URL
// 7. Update offline_routes record status = 'ready'
```

**Deno import note:** Turf.js works in Deno via npm specifier: `import * as turf from 'npm:@turf/turf';`

[Source: PRD Section 9, PRD Section 7.3.2]

### Corridor Buffer Calculation

```typescript
import * as turf from 'npm:@turf/turf';

// routePolyline: encoded Google polyline string
// Returns GeoJSON polygon of the corridor
function computeCorridor(routePolyline: string): turf.Feature<turf.Polygon> {
  const decoded = decodePolyline(routePolyline); // [[lat,lng], ...]
  const lineString = turf.lineString(decoded.map(([lat,lng]) => [lng, lat])); // GeoJSON [lng,lat]
  return turf.buffer(lineString, 0.5, { units: 'kilometers' }); // 500m buffer
}
```

### OSRM Route Graph Request

```typescript
// OSRM Table/Route endpoint: self-hosted
const response = await fetch(
  `${OSRM_ENDPOINT}/route/v1/driving/${waypoints}?steps=true&annotations=true&geometries=geojson`
);
// Extract: nodes, edges, turn instructions → serialize to compact RouteGraph
```

Route graph format (MessagePack + gzip, stored alongside MBTiles in OPFS):
```typescript
interface RouteGraph {
  nodes: Array<{ lat: number; lng: number }>;
  edges: Array<{ from: number; to: number; dist: number; dur: number; road: string }>;
  steps: Array<{ node: number; instruction: string; distance: number }>;
}
```

### Google Cloud TTS — Voice Clip Generation

```typescript
// One MP3 clip per turn instruction
// Announcement triggers: 500m before turn, 200m before turn, at turn
// So ~3 clips per turn + route start/end clips
// For a 100km route with ~20 turns: estimate 60–80 clips

const ttsResponse = await fetch(
  `https://texttospeech.googleapis.com/v1/text:synthesize?key=${GCP_TTS_KEY}`,
  {
    method: 'POST',
    body: JSON.stringify({
      input: { text: `In 500 metres, ${instruction}` },
      voice: { languageCode: 'en-IN', name: 'en-IN-Wavenet-A' },
      audioConfig: { audioEncoding: 'MP3' },
    }),
  }
);
const { audioContent } = await ttsResponse.json(); // base64 MP3
```

Cost: I believe approximately ₹0.10–₹0.50 per route download — verify at the Google Cloud TTS pricing page.

### DB Schema — New Table This Story

```sql
CREATE TABLE offline_routes (
  id                   UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id              UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  name                 TEXT NOT NULL,              -- "{origin} → {destination}"
  origin               TEXT NOT NULL,
  destination          TEXT NOT NULL,
  route_polyline       TEXT NOT NULL,              -- encoded Google polyline
  download_size_bytes  INTEGER,                    -- set on completion
  zoom_max             INTEGER NOT NULL DEFAULT 15, -- 13 (basic) | 15 (standard) | 17 (enhanced)
  status               TEXT NOT NULL DEFAULT 'pending'
                         CHECK (status IN ('pending', 'generating', 'ready', 'failed')),
  downloaded_at        TIMESTAMPTZ NOT NULL DEFAULT now(),
  expires_at           TIMESTAMPTZ NOT NULL,       -- now() + 30 days
  opfs_key             TEXT                        -- set client-side after OPFS write
);
CREATE INDEX idx_offline_routes_user_id ON offline_routes(user_id);
CREATE INDEX idx_offline_routes_expires_at ON offline_routes(expires_at);
```
