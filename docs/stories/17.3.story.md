# Story 17.3 — Group Ride: Create, Join & Live Broadcast

**Epic:** 17 — Maps, Group Rides & Offline Navigation
**Story:** 17.3
**Status:** Draft
**Created:** 2026-05-17
**Depends On:** Story 17.1 (Map embed working), Story 17.2 (Marker clustering available); **GATE: Supabase Realtime PoC from Story 17.1 AC 20 must confirm free tier is adequate before this story begins**

---

## Story

**As a** ride group leader,
**I want** to create a ride group with an invite code and see all members' live locations on the map,
**so that** I can coordinate my group without switching to WhatsApp for location sharing.

---

## Acceptance Criteria

1. - [ ] Any authenticated user can **create a ride group** (name mandatory, description optional)
2. - [ ] System generates a unique **6-character alphanumeric invite code** (e.g., `RD7K2M`)
3. - [ ] A **share link** is generated in format `motoyaar.app/join/{code}` — tapping it on mobile opens the app directly to the join screen
4. - [ ] Any authenticated user can **join a group** by entering a 6-character code manually
5. - [ ] A user can be in **only one active group** at a time — joining a new group while in an active one shows an error
6. - [ ] Group capacity is **maximum 20 members**; attempting to join a full group shows an error
7. - [ ] The group creator is the **leader**; leader role indicator is shown
8. - [ ] All group members are shown as **motorcycle icon markers** on the map, colour-coded per rider, labelled with display name
9. - [ ] Location updates are published via **Supabase Realtime Broadcast** on channel `ride-group:{groupId}` — ephemeral, NOT written to Postgres (except 30-second snapshots per AC 10)
10. - [ ] Every member's location is written to `location_snapshots` **once every 30 seconds** as an audit trail; **snapshot opt-out is the default** — snapshots are off unless user explicitly enables in Privacy Settings
11. - [ ] Location broadcast payload is minimal: `{ u, la, lo, s, h, t }` (userId, lat, lng, speed, heading, timestamp — short keys)
12. - [ ] Location update frequency: moving >30 km/h → every 3s; moving <30 km/h → every 5s; stationary >60s → every 30s; app backgrounded → every 15s (significant-changes mode)
13. - [ ] **Dead-zone filter**: location update is NOT published if movement is < 10 metres since last publish
14. - [ ] Invite codes **expire 24 hours** after group creation (default); leader can regenerate code at any time (invalidates old code)
15. - [ ] Group is in **WAITING** state after creation; members see each other on map with 15-second update intervals while waiting
16. - [ ] `pnpm typecheck` and `pnpm lint` pass

### DB Schema — New Tables This Story

```sql
CREATE TABLE ride_groups (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name        TEXT NOT NULL,
  invite_code TEXT UNIQUE NOT NULL,
  created_by  UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  status      TEXT NOT NULL DEFAULT 'waiting'
                CHECK (status IN ('waiting', 'active', 'ended')),
  max_members INTEGER NOT NULL DEFAULT 20,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  started_at  TIMESTAMPTZ,
  ended_at    TIMESTAMPTZ
);
CREATE INDEX idx_ride_groups_invite_code ON ride_groups(invite_code);
CREATE INDEX idx_ride_groups_created_by ON ride_groups(created_by);

CREATE TABLE ride_sessions (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  group_id     UUID NOT NULL REFERENCES ride_groups(id) ON DELETE CASCADE,
  user_id      UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  joined_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
  left_at      TIMESTAMPTZ,
  last_seen_at TIMESTAMPTZ,
  is_leader    BOOLEAN NOT NULL DEFAULT false,
  UNIQUE(group_id, user_id)
);
CREATE INDEX idx_ride_sessions_group_id ON ride_sessions(group_id);
CREATE INDEX idx_ride_sessions_user_id ON ride_sessions(user_id);

CREATE TABLE ride_invites (
  token       TEXT PRIMARY KEY,
  group_id    UUID NOT NULL REFERENCES ride_groups(id) ON DELETE CASCADE,
  created_by  UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  max_uses    INTEGER DEFAULT NULL,
  use_count   INTEGER NOT NULL DEFAULT 0,
  expires_at  TIMESTAMPTZ NOT NULL
);
CREATE INDEX idx_ride_invites_group_id ON ride_invites(group_id);

-- Audit trail snapshots (opt-out default = snapshots OFF unless user enables)
CREATE TABLE location_snapshots (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  session_id  UUID NOT NULL REFERENCES ride_sessions(id) ON DELETE CASCADE,
  user_id     UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  location    GEOMETRY(Point, 4326),  -- PostGIS required
  speed_kmh   REAL,
  heading     REAL,
  accuracy_m  REAL,
  recorded_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_location_snapshots_session_id ON location_snapshots(session_id);
CREATE INDEX idx_location_snapshots_recorded_at ON location_snapshots(recorded_at);
```

**PostGIS note:** Verify that `CREATE EXTENSION IF NOT EXISTS postgis;` is enabled on the Supabase project before running this migration (see PRD Open Question #2).

### RLS Policies Required

```sql
-- Members can only read ride_sessions for groups they belong to
CREATE POLICY "members_read_own_sessions"
  ON ride_sessions FOR SELECT
  USING (
    user_id = auth.uid()
    OR group_id IN (
      SELECT group_id FROM ride_sessions WHERE user_id = auth.uid()
    )
  );

-- Only leader can update ride_groups status
CREATE POLICY "leader_update_group"
  ON ride_groups FOR UPDATE
  USING (created_by = auth.uid());
```

### Realtime Broadcast Channel

```typescript
// Channel name format
const channelName = `ride-group:${groupId}`;

// Broadcast payload interface (short keys to minimise message size)
interface LocationBroadcast {
  u: string;   // userId
  la: number;  // latitude
  lo: number;  // longitude
  s: number;   // speed km/h
  h: number;   // heading degrees 0–360
  t: number;   // client timestamp ms
}

// Publisher (in rider's browser)
const channel = supabase.channel(channelName);
channel.send({
  type: 'broadcast',
  event: 'location',
  payload: { u, la, lo, s, h, t },
});

// Subscriber (all group members)
channel
  .on('broadcast', { event: 'location' }, ({ payload }) => {
    updateMemberMarker(payload);
  })
  .subscribe();
```

### Location Update Frequency Logic

```typescript
function getUpdateIntervalMs(speedKmh: number, isStationary: boolean): number {
  if (isStationary) return 30_000;   // stationary >60s
  if (speedKmh > 30) return 3_000;   // moving fast
  return 5_000;                       // moving slow
}

// Dead-zone filter
const MIN_DISTANCE_METRES = 10;
function shouldPublish(prevLat, prevLng, newLat, newLng): boolean {
  return haversineDistance(prevLat, prevLng, newLat, newLng) >= MIN_DISTANCE_METRES;
}
```

### Invite Code Generation

```typescript
function generateInviteCode(): string {
  const chars = 'ABCDEFGHJKLMNPQRSTUVWXYZ23456789'; // avoid O,0,I,1 confusion
  return Array.from(crypto.getRandomValues(new Uint8Array(6)))
    .map(b => chars[b % chars.length])
    .join('');
}
```

Store code in both `ride_groups.invite_code` and a corresponding `ride_invites` record with 24h expiry.

[Source: PRD Section 7.2.1]

### Privacy — Snapshot Opt-Out Default

The `location_snapshots` writes must be gated by user preference. Add a `snapshots_enabled` boolean to user settings (default `false`). Only write to `location_snapshots` if `user.snapshots_enabled === true`.

This also controls Supabase free tier DB size — if all users default to opt-out, snapshots table stays near-empty.

[Source: PRD Section 17 Open Question #5 — Resolved: opt-out is default, PRD Section 16 Risks — Supabase DB size]
