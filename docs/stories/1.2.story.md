### Drizzle Schema

[Source: architecture.md#9-database-schema]

Implement `src/lib/db/schema.ts` as a Drizzle schema mirroring the full SQL DDL from architecture section 9. All tables must be defined. Key points:

- Use `pgTable` from `drizzle-orm/pg-core`
- UUIDs: use `uuid('id').primaryKey().defaultRandom()`
- Timestamps: `timestamp('created_at', { withTimezone: true }).notNull().defaultNow()`
- Text enums use `.notNull()` — use Drizzle `check()` constraints to mirror SQL CHECK constraints
- JSONB columns: use `jsonb('breakdown').notNull().default([])`
- Array columns (posts.images, posts.tags): use `text('images').array().notNull().default([])`

### Full SQL DDL Reference

[Source: architecture.md#9-database-schema]

The complete SQL DDL is in architecture.md section 9. Key table highlights:

**users table:**
```sql
CREATE TABLE users (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  google_id TEXT UNIQUE NOT NULL,
  name TEXT NOT NULL,
  username TEXT UNIQUE NOT NULL,
  bio TEXT,
  profile_image_url TEXT,
  instagram_link TEXT,
  currency TEXT NOT NULL DEFAULT 'INR',
  notification_window_days INTEGER NOT NULL DEFAULT 30,
  document_storage_preference TEXT NOT NULL DEFAULT 'parse_only'
    CHECK (document_storage_preference IN ('parse_only', 'full_storage')),
  push_notifications_enabled BOOLEAN NOT NULL DEFAULT true,
  walkthrough_seen BOOLEAN NOT NULL DEFAULT false,
  status TEXT NOT NULL DEFAULT 'active'
    CHECK (status IN ('active', 'warned', 'suspended', 'banned')),
  suspended_until TIMESTAMPTZ,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

**vehicles** unique constraint: `UNIQUE(user_id, registration_number)`

**expenses** reason CHECK: `CHECK (reason IN ('Service', 'Fuel', 'Trip', 'Others'))`

**documents** status CHECK: `CHECK (status IN ('valid', 'expiring', 'expired', 'incomplete'))`

**ai_reports** status CHECK: `CHECK (status IN ('pending', 'generating', 'ready', 'failed'))`

**admin_settings** seed data:
```sql
INSERT INTO admin_settings VALUES
  ('auto_hide_report_threshold', '10', now()),
  ('default_notification_window_days', '30', now());
```

### Row Level Security (RLS)

[Source: architecture.md#database-security]

After applying the schema migration, enable RLS on all user-owned tables via Supabase Dashboard or SQL editor. RLS is defense-in-depth — the application-level auth (NextAuth session check) is the primary guard.

```sql
-- Enable RLS on all user-owned tables
ALTER TABLE users ENABLE ROW LEVEL SECURITY;
ALTER TABLE vehicles ENABLE ROW LEVEL SECURITY;
ALTER TABLE documents ENABLE ROW LEVEL SECURITY;
ALTER TABLE expenses ENABLE ROW LEVEL SECURITY;
ALTER TABLE trips ENABLE ROW LEVEL SECURITY;
ALTER TABLE posts ENABLE ROW LEVEL SECURITY;
ALTER TABLE comments ENABLE ROW LEVEL SECURITY;
ALTER TABLE post_reactions ENABLE ROW LEVEL SECURITY;
ALTER TABLE post_reports ENABLE ROW LEVEL SECURITY;
ALTER TABLE notifications ENABLE ROW LEVEL SECURITY;
ALTER TABLE vehicle_invites ENABLE ROW LEVEL SECURITY;
ALTER TABLE vehicle_access ENABLE ROW LEVEL SECURITY;
ALTER TABLE push_subscriptions ENABLE ROW LEVEL SECURITY;
ALTER TABLE ai_reports ENABLE ROW LEVEL SECURITY;
```

> For MVP, permissive RLS policies (allow service role) are acceptable since all queries go through authenticated API routes. Full per-user policies can be tightened post-MVP.

### `pgcrypto` Extension

[Source: architecture.md#9-database-schema]

The schema uses `gen_random_uuid()`. Ensure the extension is enabled in Supabase:

```sql
CREATE EXTENSION IF NOT EXISTS "pgcrypto";
```

This is typically pre-enabled on Supabase projects.
